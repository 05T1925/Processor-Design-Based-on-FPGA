# 每日协作流程

用途：规定四人小组每天如何同步目标、更新任务、记录 AI 与调试过程、提交代码和处理风险。

最后更新时间：2026-07-06

## 1. 每天开始前

- A 检查 `docs/design/task_board.md`，确认当天 P0/P1/P2 目标。
- 每位成员确认自己当天要修改的文件路径，避免越界修改。
- 如需修改公共接口，先在群里说明原因，再更新 `docs/design/interfaces.md`。
- 如需修改 ISA 或 memory map，必须由 A 确认。
- 如需修改 `soc_top`、`minisys_top`、`.xdc`，必须通知 C 和 A。

推荐上午同步内容：

| 项目 | 说明 |
|---|---|
| 昨日完成 | 说明已完成任务、commit、测试结果 |
| 今日目标 | 每人 1-3 个可验证目标 |
| 接口变更 | 是否影响其他成员 |
| 阻塞问题 | 是否需要降级或协助 |

## 2. 开发中如何更新任务状态

- 每个成员只更新自己负责的任务状态。
- 状态使用 `TODO`、`IN_PROGRESS`、`BLOCKED`、`DONE`。
- 任务进入 `DONE` 前必须有测试、截图、日志或人工检查记录。
- 接口变更完成后，同步更新 `interfaces.md`、相关 testbench 和任务看板。

## 3. 每天结束前需要提交什么

- 每天至少有一个可说明进度的 commit。
- 每位成员更新自己的任务状态。
- 有测试则提交测试结果说明或截图路径。
- 有 bug 则记录现象、原因、解决方案、验证结果。
- 有 AI 使用则当天记录到 `docs/ai_logs/ai_usage_log.md`。
- `main` 分支不能提交不能跑的版本。

推荐晚上收口流程：

| 时间段 | 动作 |
|---|---|
| 晚上前半段 | B/C/D 提交各自模块、测试结果和日志 |
| 晚上后半段 | A 做集成检查、接口检查、看板更新 |
| 当天结束 | 确认 `main` 可运行或保留稳定分支 |

## 4. AI 使用日志记录

- 每个成员只记录自己实际使用 AI 的部分。
- 不能把 AI 日志全部交给组长补。
- 每次 AI 生成或修改代码、文档、测试思路，都需要记录。
- AI 输出必须人工审阅，核心模块必须经过 xsim 或人工检查后才能合并。
- 最终报告附录会引用 `docs/ai_logs/ai_usage_log.md`。

## 5. 调试日志记录

出现 bug 时记录：

- bug 现象。
- 影响模块。
- 复现方式。
- 原因分析。
- 解决方案。
- 关联 commit。
- 波形截图路径。
- Vivado 报告路径。
- 验证结果。

## 6. Pull Request 或合并流程

1. 成员在自己的 feature 分支完成开发。
2. 更新相关文档、测试说明和 AI 日志。
3. 自查 `docs/team/review_checklist.md`。
4. 提交 commit。
5. 向 A 说明变更范围、测试结果和是否影响接口。
6. A 检查是否破坏 `main`、是否影响其他成员接口。
7. 通过后合并。

## 7. 接口冲突处理

- 先暂停继续扩散修改。
- 找到冲突接口对应文档：`interfaces.md`、`isa.md` 或 `memory_map.md`。
- 由主责成员提出修改原因。
- A 组织相关成员确认。
- 先改文档，再改 RTL，再改 testbench。
- 重新跑相关 xsim 仿真。

## 8. 进度延误与降级

| 风险 | 降级方案 |
|---|---|
| CPU basic 未跑通 | 暂停 MAC/流水线冲刺，集中修 P0 |
| 上板端口不匹配 | 以 `constraints/minisys.xdc` 和 `docs/hardware/minisys_pinout.md` 为准，先修 `minisys_top` 映射 |
| 数码管异常 | 先用 LED 显示低位结果或状态 |
| MAC 时序异常 | 报告中保留 MAC 仿真和 PPA 分析，上板演示基础 CPU |
| 流水线未完成 | 作为冲刺探索记录，不影响主线验收 |
| Vivado 数据不完整 | 保留 xsim 数据，明确缺失原因和补测计划 |

## 9. 每日最小完成标准

- 每人有任务状态更新。
- 每人有 commit、日志或可说明的阻塞记录。
- 有 AI 使用则有日志。
- 有接口变更则文档同步。
- `main` 不被不可运行版本污染。

## 10. 组员环境准备

1. 先按老师提供的安装包安装 Vivado 2018.3。
2. 组内主仿真工具统一使用 Vivado xsim。
3. ModelSim 可个人使用，但不是组内强制流程。
4. 安装路径尽量不要包含中文或空格。
5. 安装完成后截图或在群里说明环境已就绪。
6. 不要把安装包、破解文件、license 文件、大型软件包上传到 GitHub。
7. 不要把 Vivado 自动生成的临时工程文件提交到仓库。
8. 环境装好后先阅读 `README.md` 和 `docs/team/member_roles.md`，再按 `docs/design/task_board.md` 领取任务。
