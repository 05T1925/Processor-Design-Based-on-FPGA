# 项目B开发指南可行性检查

## 1. 检查结论

现有 `docs/design/项目B_vibecoding开发指南.md` 的总体方向是可行的：它围绕题目 B 的核心要求，把项目主线收束到 CPU、SoC、流水线、MAC 指令、性能量化和 PPA 分析，符合课程任务书对基础层次、进阶层次和拓展层次的要求。

但结合一周周期和组长人工确认，指南中部分目标需要重新分级，避免把所有拓展点都当成必做项。

## 2. 已人工确认的项目口径

- ISA：最终选择 RV32I 子集。
- 第一版 CPU：最终采用多周期 FSM，不把五级流水线作为第一版必做项。
- 拓展策略：不强制完成五级流水线完整冒险处理；MAC 指令与 PPA 分析可作为主要拓展成果。
- 开发环境：统一使用 Vivado 2018.3。
- 仿真主流程：统一使用 Vivado xsim，ModelSim 仅作为个人可选。
- 提交材料：报告需要包含完整 GitHub 提交历史和 AI 使用日志。
- 演示方式：第一阶段使用数码管显示结果和周期；后期进阶演示可增加 UART 输出。
- MAC 指令：允许作为自定义 ISA 扩展，采用 `rd_old` 第三读口方案。
- HALT：统一使用 `EBREAK = 0x00100073`。
- 约束文件：Minisys 官方 `.xdc` 后续由组长补充更详细资料。

## 3. 推荐目标分级

### 3.1 保底目标

保底目标必须优先完成，直接决定能否验收：

- RV32I 子集多周期 FSM 基础 CPU。
- 支持算术、逻辑、访存、分支/跳转、停止类测试指令。
- 指令存储器和数据存储器先使用 BRAM。
- 基础 memory-mapped I/O。
- 数码管显示计算结果或周期低位。
- LED 显示 running、done、error 等状态。
- Vivado 仿真、综合、实现和上板演示。

### 3.2 主要进阶目标

这些目标建议作为一周项目的主线亮点：

- 小型 SoC：CPU + BRAM memory + GPIO + seg7 + performance counter。
- 性能计数器：cycle_count、instret_count、mac_count，视进度加入 stall_count 和 flush_count。
- MAC 自定义指令。
- 普通点积程序与 MAC 点积程序对比。
- CPI、周期数、资源占用、PPA 分析。

### 3.3 冲刺目标

这些目标可以做，但不应阻塞保底系统：

- 五级流水线基础版本。
- 数据前推 forwarding。
- load-use stall。
- branch flush。
- UART 输出性能统计。

### 3.4 暂不建议作为主线

- DDR3 控制器。
- Cache。
- VGA 图形显示。
- EES-329B 的 WiFi、蓝牙、电机、触摸屏、LCD。
- 浮点运算。
- 完整 RV32I。

## 4. 对现有指南的修正建议

现有指南中“流水线冒险完整处理 + MAC 自定义指令加速 + PPA 对比分析”写得偏满。建议后续协作时采用如下口径：

> 本项目以 RV32I 子集多周期 FSM CPU 和小型 SoC 上板运行为基础，以 MAC 自定义指令和 PPA 量化分析作为主要拓展；五级流水线和完整冒险处理作为分级冲刺目标，按时间逐步实现。

这样既保留工程难度和创新性，又能保证一周内可交付。

## 5. 下一步执行原则

- 先文档定接口，再写模块。
- 先单周期或多周期 CPU 跑通，再做流水线。
- 先 BRAM，不碰 DDR3。
- 先数码管和 LED，不依赖 UART。
- 每次 AI 辅助都记录日志。
- 每个可运行版本都提交 Git，方便报告展示提交历史。
