### 2.2 🥇 主打推荐：猜数字游戏（Number Guessing Game）

#### 游戏说明

1. CPU上电后使用16位LFSR生成伪随机目标数（1~65535）
2. 玩家通过SW[15:0]拨码开关输入16位猜测值
3. 确认机制：玩家拨好开关后，等待LED稳定显示比较结果
   - **LED[15:8] (红灯)**：猜测值 > 目标值（提示"太大"）
   - **LED[7:0] (蓝灯)**：猜测值 < 目标值（提示"太小"）  
   - **全部熄灭**：猜中！
4. 数码管实时显示：
   - **左4位**：当前猜测次数（十六进制）
   - **右4位**：当前开关输入值（十六进制）
5. 猜中后LED全部闪烁3次，数码管显示最终猜测次数
6. 按复位键开始新一轮游戏

#### 指令类型全覆盖验证

| 指令类别 | 具体指令 | 游戏中的用途 |
|---|---|---|
| **算术** | ADD/SUB/ADDI/SLT/SLTU | 随机数计算、差值比较、计数累加、大小判定 |
| **逻辑** | AND/OR/XOR/ANDI/ORI/XORI | LFSR移位异或、掩码提取、LED模式编码、位操作 |
| **移位** | SLL/SRL/SRA/SLLI/SRLI/SRAI | LFSR状态移位、LED闪烁移位、功率指示 |
| **访存** | LW/SW | 存储目标数(`data_ram[0]`)、猜测历史数组(`data_ram[1:N]`)、最优成绩 |
| **跳转** | JAL/JALR | 主循环跳回、子程序调用（随机数/LED显示/数码管刷新） |
| **分支** | BEQ/BNE/BLT/BGE/BLTU/BGEU | 大小比较分支、猜中退出、超时判定、循环控制 |
| **立即数** | LUI/AUIPC | 加载外设MMIO基址(`0xFFFFFC00`/`0x10000000`) |

#### 程序结构伪代码

```
// 初始化
data_ram[0] = LFSR_random()      // 目标数
data_ram[1] = 0                   // 猜测次数计数器
data_ram[2] = 0                   // 当前猜测值

main_loop:
    // 读取开关输入
    guess = MMIO[SWITCH]          // 0xFFFF_FC10
    
    // 跳过无输入（开关全0 = 未准备好）
    if guess == 0: goto main_loop
    
    data_ram[2] = guess           // 保存当前猜测
    data_ram[1]++                 // 计数+1
    
    // 比较大小
    if guess == data_ram[0]:
        goto win                   // 猜中!
    elif guess > data_ram[0]:
        MMIO[LED] = 0xFF00        // 红灯：太大
    else:
        MMIO[LED] = 0x00FF        // 蓝灯：太小
    
    // 更新数码管显示
    seg7_value = (count << 16) | guess
    MMIO[SEG7] = seg7_value
    
    // 软件延时（或等TIMER实现后用定时器）
    call delay_loop
    
    goto main_loop

win:
    // LED闪烁3次
    for i in 0..3:
        MMIO[LED] = 0xFFFF
        call delay_long
        MMIO[LED] = 0x0000
        call delay_long
    
    // 数码管显示最终成绩
    MMIO[SEG7] = data_ram[1]      // 显示猜测次数
    
    // 停机
    ebreak

// LFSR 16位伪随机数生成器
LFSR_random:
    // 实现: x[n+1] = (x[n] >> 1) ^ (x[0] ? 0xB400 : 0)
    // 使用多项式 x^16 + x^14 + x^13 + x^11 + 1
    ...
    return random_value
```

#### 估算指标

| 指标 | 估算值 | 说明 |
|---|---|---|
| 程序总指令数 | ~200-300条 | 含LFSR子程序和显示控制 |
| 数据内存使用 | ~100字节 | 目标数 + 猜测历史 + 栈 |
| 单次猜测耗时 | ~1200周期 @100MHz ≈ 12μs | 6状态FSM × ~200条指令/轮 |
| 玩家感知延迟 | 可忽略 | 远快于人类反应时间(~200ms) |
| 最坏情况猜中 | 16次猜测（二分法） | 总数65536，每次折半 |

### 2.3 🔮 P2进阶：贪吃蛇（Snake）— 需VGA实现

#### 为什么选贪吃蛇而非俄罗斯方块

| 对比维度 | 贪吃蛇 | 俄罗斯方块 |
|---|---|---|
| 程序指令数 | ~300-500条 | ~800-1200条 |
| 核心算法 | 数组移位 + 坐标碰撞 | 旋转 + 消行 + 7种方块状态机 |
| 渲染复杂度 | 逐帧画蛇+食物 | 逐帧画方块+预览+分数 |
| 碰撞检测 | O(1)查表 | O(N)遍历 |
| VGA帧缓冲 | ~2.4KB字符模式 | ~9.6KB 320×240 1bpp |
| 适配时间 | 1-2天 | 3-5天 |

#### 最小可行方案（VGA字符模式 80×30）

```
硬件需求：
  - VGA 640×480@60Hz 时序
  - 字符ROM 8×16点阵（ASCII 0x00-0x7F）
  - 帧缓冲 80×30 = 2400字节（字符码 + 颜色属性）
  - PS/2键盘 或 4×4矩阵键盘（方向控制）

游戏结构：
  初始化:
    蛇身队列[64] ← 初始位置(40,15)
    方向 ← RIGHT
    食物 ← 随机坐标(不重叠蛇身)
    得分 ← 0
  
  主循环(每帧):
    读键盘方向 → 防止反向 →
    计算新头部 →
    碰撞检测(撞墙/撞自身) → 游戏结束
    吃到食物? → 蛇增长+新食物+得分+1 →
    移动蛇身(数组移位: tail出队, head入队) →
    写帧缓冲(清屏→画边框→画蛇→画食物→写分数) →
    等待VGA垂直同步
```

#### 估算指标

| 指标 | 估算值 | 说明 |
|---|---|---|
| 程序总指令数 | ~350-500条 | 含VGA帧缓冲操作 |
| VGA帧缓冲 | 2.4KB | 80×30字符模式 |
| 蛇身数组 | 256字节 | 最多64个坐标对 |
| 每秒帧数(FPS) | ~10-15fps (多周期FSM) | 受限于SW无DMA帧缓冲更新 |
| FPS(五级流水线) | ~30-60fps | 流水线大幅加速帧缓冲运算 |

### 2.5 🥈 备选：整数计算器（Integer Calculator）

#### 游戏说明

1. 开关分三组：SW[15:12]=操作码，SW[11:6]=操作数A，SW[5:0]=操作数B
2. 数码管实时显示：左4位=操作数A，右4位=操作数B
3. LED显示当前操作类型（颜色编码）
4. 计算结果自动显示在数码管上

#### 支持运算

| 操作码 | 运算 | 实现方式 |
|---|---|---|
| 0000 | A + B | ADD指令 |
| 0001 | A - B | SUB指令 |
| 0010 | A × B | MAC指令加速 或 软件移位乘 |
| 0011 | A / B | 软件除法（移位减） |
| 0100 | A AND B | AND指令 |
| 0101 | A OR B | OR指令 |
| 0110 | A XOR B | XOR指令 |
| 0111 | A << B | SLL指令 |
| 1000 | A >> B | SRL指令 |

#### 创新亮点

- **全面覆盖ALU**：9种运算一次性展示
- **MAC加速乘法**：硬件MAC vs 软件移位乘法，性能对比一目了然
- **可对比CPI**：不同运算类型的CP​​I可以通过性能计数器采集

## 三、性能瓶颈分析

### 3.1 架构层面瓶颈

| 瓶颈 | 当前状态 | 量化影响 | 根因 |
|---|---|---|---|
| **多周期FSM** | CPI≈4-6（取决于指令类型） | IPC仅~0.17-0.25 | 6状态串行执行，无指令重叠 |
| **组合逻辑MAC** | 32b×32b单周期 | WNS=7.2ns目前充裕 | 但若提频到200MHz可能成瓶颈 |
| **无流水线** | 指令完全串行 | 吞吐量≈16-25 MIPS @100MHz | 无指令级并行(ILP) |
| **无Cache** | BRAM单周期，无预取 | 取指和数据访问不重叠 | 多周期FSM下不重叠反而无冲突 |
| **轮询I/O** | 软件不断读SWITCH | CPU浪费>99%周期等输入 | 无中断机制 |

### 3.2 存储瓶颈

| 瓶颈 | 当前状态 | 影响 | 改善方向 |
|---|---|---|---|
| ibus/dbus分离 | 取指和数据访问不冲突 | ✅ 当前够用 | 流水线后可能产生结构冲突 |
| BRAM容量 | 32KB指令+32KB数据 | ✅ 猜数字程序<1KB | VGA帧缓冲需要额外空间 |
| dbus竞争 | 数据RAM和外设共享dbus | ⚠️ VGA刷新会持续占用dbus | 双端口BRAM或独立VGA总线 |

### 3.3 I/O瓶颈

| 瓶颈 | 当前状态 | 影响 | 改善方向 |
|---|---|---|---|
| 无定时器 | 延时靠软件循环(NOP) | 不精确 + 浪费CPU周期 | P1实现TIMER(0xFFFF_FC70) |
| 显示密度低 | LED 16bit + SEG7 8digit | 信息展示受限 | P2实现VGA |
| 输入方式单一 | 仅16位拨码开关 | 交互体验受限 | P2实现键盘 |
| 无UART调试 | 无法打印调试信息 | 开发效率低 | P2实现UART |

### 3.4 量化估算（以猜数字游戏为例）

```
程序规模:      200条指令 (约800字节)
数据规模:      100字节 (目标数+历史+栈)

执行流程分析（一次猜测循环）:
  FETCH:      1周期 × 200条  = 200周期
  DECODE:     1周期 × 200条  = 200周期
  EXECUTE:    1周期 × 200条  = 200周期
  MEMORY:     1周期 × 20条   =  20周期  (仅LW/SW)
  WRITEBACK:  1周期 × 200条  = 200周期
  总计:       约820周期 @100MHz ≈ 8.2μs

玩家反应时间: ~200ms (200,000μs)
CPU利用率:    8.2μs / 200ms ≈ 0.004%  ← 几乎完全空闲!

优化方向:
  1. 加入TIMER中断 → CPU在等待时可执行其他任务
  2. 五级流水线 → CPI从4-6降到~1.2 → 8.2μs → ~2μs
  3. 但实际瓶颈是玩家速度，不是CPU速度
```

### 3.5 关键发现：交互式程序的瓶颈在人，不在CPU

对于猜数字/反应测试这类交互式游戏：
- CPU计算耗时（~10μs）远小于人类反应时间（~200ms）
- 真正瓶颈是**I/O轮询效率**，而非计算吞吐量
- **最重要的优化是实现TIMER**：将轮询改为定时采样，释放CPU

对于图形类游戏（贪吃蛇/打砖块，如果做VGA）：
- VGA帧缓冲更新是计算密集操作
- 多周期FSM下帧率可能只有10-15fps
- **五级流水线的价值在图形渲染中才能真正体现**

## 五、创新点总结

### 5.1 本项目独有的创新优势

| 创新点 | 说明 | 区别于参考仓库 |
|---|---|---|
| **MAC自定义指令** | opcode=0001011, funct7=0000001, `rd_new=rd_old+rs1*rs2` | 五个参考仓库均无此实现 |
| **三读口regfile** | 为MAC指令特别设计第三读端口 | 标准RV32I只需要二读口 |
| **CPU_MODE参数化** | 同一工程5种CPU架构一键切换 | 同级项目中未见类似设计 |
| **统一总线SoC** | ibus+dbus+16外设地址槽+二级译码 | SEU参考的升级版 |
| **性能计数器** | cycle/instret/mac三计数器+MMIO暴露 | 可从软件读取自身性能 |
